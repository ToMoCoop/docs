ClarusWMS extends its warehouse management platform with an **agent platform** built on **AWS Bedrock AgentCore** — a Strands-based agent runtime, the MCP gateways in front of it, the tools it can call, and the session and authentication machinery around it. This section documents that platform.

Several **clients** drive the agent through natural language, all authenticating against the same **Cognito user pool**:

- **External MCP clients** like Claude Desktop or ChatGPT connect through a public MCP gateway to query warehouse data — a read-only API for AI assistants.
- **The [AI App](/architecture/ai-app/overview)** — the Clarus web and embedded chat experience — drives the runtime through a private gateway with full tool access (CSV exports, file analysis, and more).
- **[AI Messaging](/architecture/ai-messaging/overview)** relays Slack and WhatsApp conversations to the same runtime.

Other internal services — such as backend automations — drive the runtime the same way; the pages in this section describe the platform from the perspective of any client.

A **CloudFront distribution** provides a unified custom domain with path-based routing and Lambda@Edge functions that bridge Cognito's OAuth implementation with the MCP protocol's requirements. The platform runs in **us-east-1** as several required services aren't yet available in our preferred eu-west-2 region.

<Note>
The agent platform's code lives in **ClarusWMS_Agents**; its clients live in **ClarusWMS_AI** (the AI App) and **ClarusWMS_AI_Messaging**; and **ClarusWMS_OPS** contains the Terraform infrastructure for all of them.
</Note>

## How a Request Flows

Both access paths — external MCP clients and internal clients like the AI App — share the same CloudFront distribution and authenticate with Cognito JWTs, but they diverge at routing. CloudFront's [path-based behaviors](/architecture/agents/infrastructure#cloudfront-routing-table) send `/mcp*` traffic to the public gateway and `/agent*` traffic to the Bedrock data plane, while the private gateway is only reachable from the agent runtime internally.

<Tabs>
  <Tab title="External MCP Client">
    **Scenario**: Claude Desktop queries stock levels for a warehouse tenant.

    <Steps>
      <Step title="Authenticate via OAuth">
        The MCP client runs a PKCE authorization code flow against [Cognito](/architecture/agents/authentication) — discovering endpoints via `/.well-known/openid-configuration`, registering via Dynamic Client Registration, and exchanging an authorization code for a JWT access token.
      </Step>
      <Step title="Send MCP request to CloudFront">
        The client sends a request to `/mcp/<tenant>`. A [CloudFront Function](/architecture/agents/authentication#multi-tenant-routing) extracts the tenant segment from the URL path, rewrites the URL, and sets the `X-Clarus-Subdomain` header.
      </Step>
      <Step title="Public MCP Gateway validates and routes">
        The [public gateway](/architecture/agents/gateways#public-mcp-gateway) validates the JWT's `client_id` claim against its allowed clients list. The [MCP Interceptor](/architecture/agents/infrastructure#mcp-interceptor) caches the token in DynamoDB keyed by `requestId`, then the gateway invokes the API Lambda.
      </Step>
      <Step title="API Lambda executes the tool">
        The [API Lambda](/architecture/agents/tools#api-lambda-clarus-mcp-api) retrieves the cached JWT from DynamoDB and executes the requested tool. Not all tools call the WMS backend — discovery and schema inspection tools like `api_discover`, `graphql_details`, and `rest_details` resolve from cached schemas in S3. Execution tools like `graphql_query` and `rest_query` do call the backend, using the user's own token for authorization.
      </Step>
      <Step title="Response returns through CloudFront">
        The tool result flows back through the gateway. [Lambda@Edge](/architecture/agents/infrastructure) rewrites any `WWW-Authenticate` headers to reference the custom domain rather than the raw gateway domain.
      </Step>
    </Steps>
  </Tab>

  <Tab title="AI App User">
    **Scenario**: A user asks "Show me today's picking performance and export it as CSV."

    <Steps>
      <Step title="User already authenticated">
        The user holds a Cognito JWT — obtained via [NextAuth](/architecture/agents/authentication) in standalone mode or from the parent app via [postMessage](/architecture/ai-app/overview#embedded-mode-iframe-postmessage-protocol) in embedded mode.
      </Step>
      <Step title="Message sent to Agent Runtime">
        The AI App's [SSE bridge](/architecture/ai-app/overview#api-routes-reference) (`POST /api/chat`) forwards the message to `/agent/invocations` through CloudFront. A [Lambda@Edge function](/architecture/agents/runtime#entry-point-flow) rewrites the path to the Bedrock data plane's internal format and signs the request with SigV4.
      </Step>
      <Step title="Agent reasons and calls tools">
        The [Strands agent](/architecture/agents/runtime) receives the message, decides it needs two tools — a GraphQL query for picking data and a CSV export — and makes MCP tool calls to the [private gateway](/architecture/agents/gateways#private-mcp-gateway), forwarding the user's JWT.
      </Step>
      <Step title="Private MCP Gateway dispatches">
        The [private gateway](/architecture/agents/gateways#private-mcp-gateway) validates the JWT and routes each tool call to the correct Lambda target. The [MCP Interceptor](/architecture/agents/infrastructure#mcp-interceptor) caches the token for each request, just as with the public path.
      </Step>
      <Step title="Tool Lambdas call WMS backend">
        The [API Lambda](/architecture/agents/tools#api-lambda-clarus-mcp-api) queries picking performance via GraphQL (storing the result as an NDJSON dataset in S3), then the [Utils Lambda](/architecture/agents/tools#utils-lambda-clarus-mcp-utils) generates a CSV from that dataset and returns a presigned S3 download URL.
      </Step>
      <Step title="Streaming response reaches user">
        Results stream back through the agent runtime → CloudFront → [SSE bridge](/architecture/ai-app/overview#sse-client-protocol) → browser. The AI App renders the picking data as text and the CSV as a [download card](/architecture/ai-app/overview#file-download-cards) with a live expiry countdown.
      </Step>
    </Steps>
  </Tab>
</Tabs>

<Info>
  **Auth propagation**: In both paths, the user's Cognito JWT flows end-to-end from client to WMS backend — there are no service accounts or shared credentials in the request path. The [Token Forwarding](/architecture/agents/authentication) mechanism (Interceptor → DynamoDB → Lambda) bridges the gap where MCP gateways validate but don't forward the token. See the [Authentication](/architecture/agents/authentication) page for the full detail.
</Info>

## AWS Account Architecture

Each environment runs in a **dedicated AWS account**. A shared Pipelines account triggers builds via CodePipeline V2, and CodeBuild deploys cross-account using IAM roles. Terraform manages resources in each account via an organization-level access role.
