Authentication in the AI layer serves two distinct client types — external MCP clients and the frontend application — while multi-tenant routing ensures every request is scoped to the correct warehouse tenant.

## Multi-Tenant Routing

ClarusWMS is a multi-tenant platform. When an AI request arrives, the system needs to know which tenant's data to query. The tenant extraction mechanism enables this through a single MCP gateway:

<Steps>
  <Step title="CloudFront Function intercepts the request">
    A lightweight CloudFront Function intercepts requests to `/mcp/<tenant>/*`
  </Step>
  <Step title="Tenant extracted from URL path">
    The tenant identifier is extracted from the URL — for example, `/mcp/acme/` yields tenant `acme`
  </Step>
  <Step title="URL rewritten">
    The URL is rewritten to `/mcp/` (stripping the tenant segment)
  </Step>
  <Step title="Subdomain header set">
    An `X-Clarus-Subdomain: acme` header is set on the request
  </Step>
  <Step title="Header flows through the stack">
    The header flows through CloudFront → Gateway → Interceptor → Lambda
  </Step>
  <Step title="Lambda scopes API calls">
    The target Lambda uses `X-Clarus-Subdomain` to scope all WMS API calls to that tenant
  </Step>
</Steps>

Direct access without the tenant in the path is also supported when `X-Clarus-Subdomain` is provided as a header directly — this is how the frontend app works.

  <Tab title="Frontend App">
    The frontend doesn't connect to Agent Core directly. A Next.js API route (`POST /api/chat`) acts as a server-side SSE bridge that handles auth token management, subdomain resolution, and stream re-serialization before forwarding requests through CloudFront to the agent runtime:

    <Frame>
      <img src="/images/auth-frontend-flow.drawio.svg" alt="Frontend app authentication flow from browser through Next.js SSE bridge, CloudFront, Lambda@Edge, Agent Runtime, to Private MCP Gateway" zoom />
    </Frame>

    <sub><a href="/images/auth-frontend-flow.drawio.svg" download="auth-frontend-flow.drawio.svg">Download SVG</a> · <a href="/images/auth-frontend-flow.drawio" download="auth-frontend-flow.drawio">Download draw.io source</a></sub>

    The SSE bridge serves multiple purposes: auth token management (auto-refresh on 401), dual auth support (NextAuth vs Bearer header), subdomain resolution (cookie vs request body), session tracking, keep-alive pings every 15 seconds, and stream re-serialization from Agent Core's raw format to a simplified client protocol. The Lambda@Edge function translates the friendly `/agent/*` path into the Bedrock data plane's internal format, and the runtime validates the JWT independently before reusing the same token for MCP tool calls through the private gateway.

    **Headers forwarded to Agent Core:**

    | Header | Value | Purpose |
    |--------|-------|---------|
    | `Authorization` | `Bearer <access_token>` | Cognito JWT |
    | `X-Amzn-Bedrock-AgentCore-Runtime-Session-Id` | UUID v4 | Session identifier |
    | `X-Amzn-Bedrock-AgentCore-Runtime-Custom-Subdomain` | Tenant subdomain | Tenant routing — the only custom header format AgentCore passes through to the runtime |
    | `X-Clarus-Subdomain` | Tenant subdomain | Tenant routing — sent for consistency, though only the Bedrock-prefixed header reaches the runtime |

  </Tab>
</Tabs>

### Internal Auth Mechanisms

<Accordion title="Token Forwarding (Interceptor → DynamoDB)">
  Since MCP gateways validate JWTs but don't forward them to target Lambdas, the interceptor bridges this gap using DynamoDB:

  <Frame>
    <img src="/images/auth-token-forwarding.drawio.svg" alt="Token forwarding sequence diagram showing interceptor caching bearer token in DynamoDB with 60s TTL" zoom />
  </Frame>

  <sub><a href="/images/auth-token-forwarding.drawio.svg" download="auth-token-forwarding.drawio.svg">Download SVG</a> · <a href="/images/auth-token-forwarding.drawio" download="auth-token-forwarding.drawio">Download draw.io source</a></sub>

  The 60-second TTL ensures tokens don't linger in DynamoDB. The interceptor always returns a passthrough response — a failed cache write never blocks the request.
</Accordion>

<Accordion title="Dual Auth Decision (Standalone vs Embedded)">
  The SSE bridge supports two authentication modes. When a request arrives, it checks for an `Authorization` header to determine the mode:

  <Frame>
    <img src="/images/auth-dual-auth.drawio.svg" alt="Dual auth decision flowchart showing standalone vs embedded mode branching and token refresh paths" zoom />
  </Frame>

  <sub><a href="/images/auth-dual-auth.drawio.svg" download="auth-dual-auth.drawio.svg">Download SVG</a> · <a href="/images/auth-dual-auth.drawio" download="auth-dual-auth.drawio">Download draw.io source</a></sub>

  - **Standalone mode** uses NextAuth v5 with Cognito OAuth. Tokens are stored in an encrypted session cookie and auto-refresh when within 5 minutes of expiry.
  - **Embedded mode** receives tokens from the parent app via `postMessage`. The JWT is decoded server-side for the `sub` claim but not verified — Agent Core validates it.
</Accordion>

## Key Authentication Details

- **Region**: AI infrastructure in **us-east-1** (service availability); Cognito user pool in **eu-west-2** (shared organisational resource)
- **JWT claim**: Validation uses `client_id` claim (not `aud`) — this is Cognito access token behavior
- **OAuth scopes**: `openid`, `claruswms/full`
- **PKCE**: S256 method required for authorization code flow
- **Public gateway**: Restricted to external MCP client IDs only
- **Private gateway**: Broader allowed clients (frontend app, service accounts, external clients)
- **Agent runtime**: Own allowed clients list, same OAuth discovery URL
- **RFC 8707**: `resource` parameter stripped by Lambda@Edge because Cognito doesn't support it
- **WWW-Authenticate rewrite**: Lambda@Edge rewrites response headers to use custom domain instead of raw Bedrock gateway domain
