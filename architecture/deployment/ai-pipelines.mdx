AI deployments span three repositories, all on the same shared cross-account pipeline model: **ClarusWMS_Agents** (the agent platform — Lambda tools, the agent runtime, interceptor, and schema parsers), **ClarusWMS_AI** (the [AI App](/architecture/ai-app/overview), deployed as a container on ECS), and **ClarusWMS_AI_Messaging** (the [AI Messaging](/architecture/ai-messaging/overview) relay and its subgraph).

## Agent Pipelines

All agent code lives in the **ClarusWMS_Agents** repository. Five separate pipelines handle different components, triggered by file path filters on the `main` branch.

### Source Directories

| Directory | Produces | Description |
|---|---|---|
| `clarus-mcp-api/` | `clarus-mcp-api.zip` | API Lambda — GraphQL/REST operations against the WMS |
| `clarus-mcp-utils/` | `clarus-mcp-utils.zip` | Utils Lambda — CSV generation, internal utilities |
| `mcp-interceptor/` | `mcp-interceptor.zip` | Interceptor Lambda — JWT token caching |
| `agent-runtime/` | `clarus_agent/deployment.zip` | Agent runtime — Strands Python agent |
| `api-parser/` | Schema files | GraphQL/REST schema parsing scripts |
| `scripts/generators/generate-tool-schema-api.mjs` | `clarus-mcp-api-tools.json` | Tool schema for API Lambda |
| `scripts/generators/generate-tool-schema-utils.mjs` | `clarus-mcp-utils-tools.json` | Tool schema for Utils Lambda |

### Pipeline Architecture

Each pipeline follows the same pattern: source change triggers a build in the shared Build account, which deploys to staging, waits for manual approval, then deploys to production.

| Pipeline | Trigger paths |

### Deployment Steps

<Steps>
  <Step title="Build & Package">
    CodeBuild produces a zip artifact from the source directory.
  </Step>
  <Step title="Deploy to S3">
    The zip is deployed to the S3 agents bucket in the target account via a cross-account IAM role.
  </Step>
  <Step title="Update Lambda">
    Lambda function code is updated using the new zip from S3.
  </Step>
  <Step title="Publish & Alias">
    A new Lambda version is published and the `live` alias is updated to point to it.
  </Step>
  <Step title="Sync Gateway Targets">
    The tool schema is regenerated and the gateway target configuration is updated to reflect any tool changes.
  </Step>
</Steps>

<Note>
The pipeline and build accounts are **shared infrastructure** used by all ClarusWMS services. They are not AI-specific — the same CodePipeline and CodeBuild setup handles deployments for other services as well.
</Note>

## AI App Pipeline

The [AI App](/architecture/ai-app/overview) (Next.js) is deployed as a Docker container on ECS from the **ClarusWMS_AI** repository, separate from the Lambda pipelines above.

### Pipeline Stages

<Steps>
  <Step title="Source">
    ClarusWMS_AI `main` branch via CodeStar Connection triggers the pipeline.
  </Step>
  <Step title="Build">
    ARM64 Docker build in the build account. CodeBuild project `ai-app-build` uses the `amazonlinux2-aarch64-standard:3.0` image with privileged mode enabled. Pushes to ECR `claruswms-ai:latest`. 15-minute timeout.
  </Step>
  <Step title="Tag Staging">
    ECR manifest re-tag from `latest` to `staging` using `batch-get-image` + `put-image` — no Docker pull required. Outputs `imagedefinitions.json` for the deploy action.
  </Step>
  <Step title="Deploy Staging">
    CodePipeline ECS deploy action, deploying cross-account into the target environment's account.
  </Step>
  <Step title="Manual Approval">
    Deployment pauses for manual approval before promoting to production.
  </Step>
  <Step title="Tag Production">
    ECR manifest re-tag from `latest` to `production` using the same manifest copy technique.
  </Step>
  <Step title="Deploy Production">
    ECS deploy to the production account.
  </Step>
</Steps>

### Image Promotion Model

A single Docker build produces one image. Rather than rebuilding for each environment, the image is promoted via ECR tag manipulation:

1. **Build** pushes `claruswms-ai:latest`
2. **Staging** re-tags the manifest as `staging` (no rebuild, no pull)
3. **Production** re-tags the same manifest as `production`

This guarantees identical image content across all environments — the image digest never changes. The build runs once, on ARM64, in the shared build account; staging and production deploys re-tag the same manifest rather than rebuilding.

## AI Messaging Pipelines

[AI Messaging](/architecture/ai-messaging/overview) deploys from the **ClarusWMS_AI_Messaging** repository on the same shared model, in two strands:

<Steps>
  <Step title="Relay Lambdas">
    The relay's Lambda code is built into bundles, uploaded to the target account, and rolled out by updating each function's code — the same build-and-promote shape as the agent Lambda pipelines. Terraform provisions the functions with placeholder code, so the pipeline owns the running code.
  </Step>
  <Step title="Subgraph">
    The [subgraph](/architecture/ai-messaging/subgraph) image is built and pushed to ECR, then deployed to ECS. Around the deploy, the shared PostGraphile [migrate and introspect](/architecture/gateway/postgraphile-subgraph#migrate-and-introspect-codebuild) jobs apply database migrations and publish the updated SDL to the supergraph via the [publish pipeline](/architecture/gateway/publish-pipeline).
  </Step>
</Steps>

Both strands deploy cross-account into each environment's account, with promotion to production gated by manual approval — identical to the agent and AI App pipelines above.
